# Архитектура репозитория `okx-hft-executor`

Документ фиксирует системные допущения, границы ответственности и ключевые идеи, которые должны сохраняться при росте кодовой базы.

**Практический вход для новых участников:** как установить зависимости, настроить `.env` и запустить процесс — [getting_started.md](getting_started.md).

## 1. Executor — это execution core, а не «отправитель ордеров»

Исполнительный контур отвечает за **полный цикл принятия торгового намерения и доведения его до согласованного состояния с биржей и локальным ledger**:

- перевод доменного намерения в допустимые заявки с учётом режима рынка и политики исполнения;
- управление жизненным циклом заявок (создание, замена, отмена) и обработка отказов API;
- сопровождение позиции и выходов согласно политике;
- устойчивость к обрывам каналов, дубликатам событий и рестартам;
- **reconciliation**: периодическая и событийная сверка локального состояния с биржей;
- непрерывная запись журнала решений и исходов для последующего анализа и аудита.

Отправка HTTP/WebSocket-запроса — лишь один шаг в этой цепочке. Без risk-слоя, state machine, учёта и восстановления сервис не является production-grade execution core.

## 2. Почему strategy, execution, risk, persistence и exchange разделены

| Блок | Роль |
|------|------|
| **strategy** | Генерация или приём сигналов и правил входа/выхода **без** знания о транспорте к бирже и без прямых побочных эффектов на ордера. |
| **execution** | Оркестрация намерений → ордеров → позиции; state machine; политика исполнения; взаимодействие с risk и exchange через чёткие контракты. |
| **risk** | Единое место для лимитов, guard-ов и kill switch до и во время торговли; исполнение не должно «тихо» обходить risk. |
| **exchange** | Изоляция протокола OKX (и в будущем других площадок): подпись, лимиты, нормализация DTO → домен. |
| **persistence** | Долговечность журнала, снимков состояния, идемпотентность восстановления; домен не зависит от конкретной СУБД. |

Разделение снижает связанность, упрощает тестирование (paper/replay подменяют exchange), позволяет менять стратегию без переписывания транспорта и наоборот.

## 3. State reconciliation и почему он обязателен

Локальное состояние (открытые ордера, позиция, последние fill) **всегда** может расходиться с биржей из-за:

- потери или задержки WebSocket-сообщений;
- частичного исполнения и последующих изменений статуса на стороне биржи;
- повторной доставки событий;
- сбоя процесса между отправкой заявки и записью результата.

**Reconciliation** — регулярная и триггерная сверка «локальный ledger / кэш состояния» с источником истины (REST snapshot, при необходимости — детализация по ордерам). Без неё система не может гарантировать корректное поведение после рестарта или сетевых аномалий.

Подробнее: [reconciliation.md](reconciliation.md).

## 4. Локальный торговый ledger при наличии данных на бирже

Биржа хранит **свою** историю и снимки; исполнительному контуру нужен **собственный** append-only (или эквивалентный) журнал и производные представления, потому что:

- нужна **идемпотентность** и воспроизводимость решений (что именно было разрешено risk, что ушло в API, в каком порядке);
- при сбое важна **атомарность** записи намерения и последующих событий в одной транзакции приложения;
- аналитика PnL, slippage и качества исполнения строится на **единой модели** сигнал → ордер → fill → позиция;
- внешний API биржи может быть временно недоступен — локальный журнал остаётся опорой для recovery и аудита.

Биржа — источник истины для **факта** исполнения на площадке; локальный ledger — источник истины для **решений и интерпретации** в рамках данного сервиса.

## 5. Режимы: live, paper, replay

Кратко; детали: [runtime_modes.md](runtime_modes.md).

- **live**: реальные ключи, реальные заявки, полный risk и observability.
- **paper**: те же пути кода, подмена или эмуляция exchange-транспорта / демо-контура, без боевого капитала (или с изолированным счётом).
- **replay**: детерминированная подача записанных рыночных и торговых событий для регрессии логики исполнения и стратегии.

Общая **доменная** и **execution**-логика не должна ветвиться «if paper» по всему коду; режим выбирается через реализации портов (exchange, clock, источник рыночных данных).

## 6. Классы сбоев, которые контур обязан переживать

| Сценарий | Ожидание |
|----------|----------|
| Обрыв WebSocket | Переподключение, backoff, после стабилизации — reconciliation. |
| Дубликаты событий | Идемпотентная обработка по ключам (order_id, exec_id, event id). |
| Частичные исполнения | Обновление остатков заявки, корректный переход state machine. |
| Устаревшие рыночные данные | Guard-ы в risk/strategy (stale threshold), отказ от агрессивного исполнения при нарушении SLA данных. |
| Отказы API / reject | Маппинг в доменные ошибки, политика retry, алерты, без «молчаливого» рассинхрона. |
| Рестарт | Восстановление из persistence + обязательная сверка с биржей. |
| Расхождение локального и биржевого состояния | Политика в приоритете биржи с фиксацией инцидента; корректировка локального состояния и журналирование. |

## 7. Observability и kill switch

**Observability** (структурированные логи, метрики, готовность к трассировке) — не украшение: без неё невозможно расследовать инциденты в субсекундном контуре и доказывать корректность поведения.

**Kill switch** (глобальный и пер-инструмент/пер-стратегия) должен:

- останавливать новое исполнение и/или инициировать flatten по политике;
- быть доступен из control plane и из автоматических триггеров risk;
- иметь минимальную цепочку вызовов (без прохождения через «тяжёлые» зависимости).

## 8. Границы репозитория

**Внутри executor-репозитория:**

- приём/генерация сигналов и правила стратегии (как библиотека/модули сервиса);
- исполнение, risk, учёт, persistence, интеграция с OKX, observability, control hooks.

**Обычно вне этого репозитория (или отдельные сервисы/репозитории):**

- долгосрочное хранилище «сырых» тиков всего рынка (если не является целью данного сервиса);
- ML training pipeline;
- корпоративный billing / ERP;
- глобальный оркестратор портфеля уровня фонда (executor получает уже согласованные лимиты и сигналы).

Границы можно смещать осознанно; документ следует обновлять.

## 9. Точки расширения с первого дня

- Порт **ExchangeClient** (REST/WS) — смена реализации для paper/live и будущих бирж.
- **Execution policy** (алгоритмы выставления, перестановка, частичное снятие) — отдельные стратегии от ядра state machine.
- **Risk checks** как регистрируемые правила (pipeline), а не монолит.
- **Repositories / Unit of Work** — подмена хранилища без изменения домена.
- **Event bus / domain events** — для связи блоков без жёстких импортов (по мере роста).
- **Clock** — инъекция времени для replay и тестов.

## 10. Минимальный путь развития

См. детализацию в [roadmap.md](roadmap.md).

- **Фаза 1 — basic live executor**: один инструмент, прямой путь signal → order, базовый журнал, простой REST OKX.
- **Фаза 2 — resilient executor**: полноценный WS, reconciliation, идемпотентность, kill switch, метрики.
- **Фаза 3 — mature execution platform**: несколько стратегий и политик, paper/replay, расширенный учёт PnL/fees/funding, multi-exchange readiness.

## Дополнительный блок: `control/`

Control plane (HTTP или иной интерфейс) для health, паузы, принудительного flatten и операционных действий вынесен отдельно от `app/`, чтобы ядро цикла не тащило веб-стек как обязательную зависимость (опциональные extra в `pyproject.toml`).
