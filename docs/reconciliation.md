# Reconciliation (сверка состояния)

## Зачем нужен reconciliation

Исполнительный контур держит **локальное представление** открытых заявок, позиций и последних исполнений для быстрых решений и журнала. Любой сбой сети, процесса или дублирование событий приводит к **расхождению** с фактическим состоянием на бирже.

Без регулярной сверки:

- после рестарта возможны двойные заявки или «забытые» открытые ордера;
- частичное исполнение не отразится в risk и accounting;
- kill switch и лимиты будут опираться на ложные данные.

Reconciliation переводит систему в состояние **согласованное с приоритетом биржи** (exchange as source of truth for market facts), с фиксацией расхождений в журнале.

## Что сверяем

Типичный минимальный набор (конкретика зависит от рынка и API):

- список **открытых ордеров** и их параметры (цена, количество, статус);
- **позиция** по инструменту и режиму торговли;
- последние **fills** / сделки за скользящее окно (для догонки пропусков в WS);
- баланс/маржа — если влияет на допуск заявок (через risk).

Локально сверяются те же сущности в `domain` / кэше `execution`, восстановленные из `persistence` при старте.

## Сценарии рассинхрона

| Сценарий | Пример |
|----------|--------|
| Потеря WS-сообщения | Fill был на бирже, локально заявка ещё «open». |
| Дубликат события | Двойной учёт исполнения без идемпотентности. |
| Сбой между send и ack | Локально «нет ордера», на бирже ордер жив. |
| Ручное вмешательство | Ордер отменён в UI биржи. |
| Частичное исполнение + expire | Локальный остаток не обновлён. |
| Рестарт на полпути | Журнал записан частично; состояние неполное. |

## Поведение executor при расхождении

Общие правила (детали реализации — в коде `execution.reconciliation`):

1. **Приоритет данных биржи** для факта открытых заявок и позиции после успешного snapshot.
2. **Корректировка локального состояния** под результат сверки; генерация доменных событий «коррекция» для accounting и журнала.
3. **Никаких скрытых молчаливых правок** — расхождение логируется (уровень warn/error по severity), метрика инкрементируется, при крупных отклонениях — алерт.
4. Если сверка **невозможна** (API недоступен): политика «осторожного» режима — удержание новых агрессивных заявок, опора на risk и kill switch; повтор сверки с backoff.
5. После коррекции — **идемпотентная** догонка недостающих fills при необходимости (по списку trade/exec id).

## Триггеры запуска

- **Периодический** таймер (интервал из конфигурации).
- **После переподключения** WebSocket.
- **После старта процесса** (cold start) — обязательный полный snapshot критичных сущностей.
- **По запросу** оператора через control plane (будущее расширение).

## Связь с тестами

Сценарии reconciliation и recovery должны покрываться **integration** и **replay** тестами с заранее записанными ответами API и потоками WS.
