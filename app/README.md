# Пакет `app`

Запуск процесса из корня репозитория и настройка окружения: [docs/getting_started.md](../docs/getting_started.md).

## Назначение блока

Точка входа процесса, сборка графа зависимостей (bootstrap) и высокоуровневая **оркестрация** жизненного цикла исполнителя: запуск, остановка, координация asyncio-задач (или аналога), делегирование в `execution` и смежные слои.

## Почему он существует

Без выделенного слоя приложения конфигурация, логирование и run-loop смешиваются с бизнес-логикой, что усложняет тестирование и деплой. `app` фиксирует **composition root**: где создаются реализации портов и как они связываются.

## Основные обязанности

- чтение конфигурации окружения;
- инициализация observability (логи, метрики);
- создание экземпляров exchange, persistence, risk, execution;
- запуск главного цикла и корректное завершение (graceful shutdown).

## Что здесь должно находиться

- `main.py` — CLI / process entry;
- `bootstrap.py` — фабрика зависимостей и режимов (live/paper/replay);
- `orchestrator.py` — координация задач без детальной торговой логики.

## Что здесь находиться не должно

- правила стратегии и расчёт сигналов;
- детали подписи OKX или парсинга JSON ответов;
- SQL-запросы и схемы таблиц;
- тяжёлая бизнес-логика risk (только вызовы готовых компонентов).

## Основные сущности / модули

| Модуль | Роль |
|--------|------|
| `main` | Аргументы, вызов bootstrap, запуск. |
| `bootstrap` | Сборка графа по `config` и `RuntimeMode`. |
| `orchestrator` | Управление задачами цикла (данные, исполнение, сверка). |

## Связи с другими блоками

- **config** — источник настроек.
- **execution**, **exchange**, **risk**, **persistence**, **observability** — подключаются из bootstrap.
- **control** — опционально поднимается параллельно как отдельная задача.

## Примеры будущего расширения

- несколько процессов (reader / executor) с IPC;
- plug-in загрузка стратегий по имени из конфига;
- интеграция с внешним service mesh / k8s probes через `control`.
