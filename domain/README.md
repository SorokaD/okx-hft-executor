# Пакет `domain`

## Назначение блока

**Доменный слой**: сущности, перечисления, события и value objects торгового контура **без** зависимостей от HTTP, WebSocket, ORM и конкретной биржи.

## Почему он существует

Domain-first подход фиксирует язык системы: что такое сигнал, заявка, исполнение, позиция, PnL-снимок, риск-событие. Остальные пакеты адаптируют внешний мир к этим типам, а не наоборот.

## Основные обязанности

- определить стабильные типы и инварианты;
- обеспечить сериализуемость для журнала и replay;
- минимизировать «протекание» wire-форматов OKX внутрь моделей (нормализация в `exchange`).

## Что здесь должно находиться

- `models/` — агрегаты и сущности (signal, order, fill, position, …);
- `enums/` — стороны, типы заявок, статусы;
- `events/` — доменные события для связи слоёв;
- `value_objects/` — деньги, количество, идентификаторы инструментов.

## Что здесь находиться не должно

- клиенты `requests` / `aiohttp`, парсинг JSON OKX;
- SQLAlchemy модели (они — в `persistence.storage_models`, если нужны);
- расчёт конкретной стратегии входа;
- прямые вызовы risk API биржи.

## Основные сущности / модули

См. `domain/models/*.py` — каркас сущностей; по мере роста добавляются поля и инварианты.

## Связи с другими блоками

- **strategy** производит/нормализует сигналы в типы `domain`.
- **execution** управляет жизненным циклом `Order`, `Position`.
- **exchange** маппит ответы API → `domain`.
- **persistence** сохраняет сериализованные доменные события/снимки.
- **accounting** строит производные представления (PnL) над фактами домена.

## Примеры будущего расширения

- разделение `InstrumentId` на spot/swap с явным режимом маржи;
- версионирование схем событий для долгого журнала;
- доменные ошибки (`DomainError`) с кодами для API control plane.
