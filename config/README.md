# Пакет `config`

## Назначение блока

Централизованные **настройки** приложения: переменные окружения, профили, лимиты по умолчанию и типизированный доступ к ним для всех слоёв.

## Почему он существует

Разбросанные `os.getenv` делают конфигурацию непрозрачной и небезопасной. Явная модель настроек (pydantic-settings) даёт валидацию, документацию полей и единую точку изменения при деплое.

## Основные обязанности

- загрузка и валидация env / файлов;
- разделение общих и окружение-специфичных параметров;
- описание режима времени выполнения (live/paper/replay).

## Что здесь должно находиться

- `settings.py` — основная модель `Settings`;
- `limits.py` — числовые лимиты по умолчанию (или ссылки на них из Settings).

## Что здесь находиться не должно

- бизнес-правила risk (только **пороговые значения** из конфига);
- секреты в коде (только чтение из env / secret store);
- логика подключения к БД (только строки подключения / DSN).

## Основные сущности / модули

| Модуль | Роль |
|--------|------|
| `settings` | Pydantic `BaseSettings`, `get_settings()`. |
| `limits` | Константы/модели лимитов для risk (каркас). |

## Связи с другими блоками

- **app.bootstrap** — потребитель всех настроек.
- **risk**, **exchange**, **observability** — получают параметры через конструкторы / контекст, а не импортируют глобали кроме фабрик.

## Примеры будущего расширения

- отдельные файлы `production.yaml` / `development.yaml` с merge;
- динамическое обновление безопасных флагов (feature flags) с подпиской;
- валидация согласованности лимитов (например max_notional < account ceiling).
